# 要件定義 v6.2: KPの外部脳

v5仕様の完全実装 (2026-06-13) をユーザーが実際に使用した結果、「使い物にならない。便利でなく負担だけが増える。シナリオを構築できない (ルート設計ができない)」と判定された。本書はその失敗分析と要件の再定義である。**v5とその実装は判断基準として無効**。コードは凍結し、再利用は概念検証後に判断する。

> 改訂履歴: v6 (ユーザー対話の一次反映) → v6.1 (自己検討+敵対的レビュー統合。所有権モデル・条件参照機構・大筋の中身を確定) → **v6.2** (検証ケース1「悪霊の家」の結果を反映: イベント内蔵状態・スロット自動生成・否定リンク・執筆ガイドライン。docs/plans/2026-06-13-v6-validation-case1.md 参照)

---

## 1. v5実装の失敗分析

| 失敗 | 内容 |
|---|---|
| モデル化対象の誤り | ペトリネットは「世界の状態」をモデル化したが、KPに必要なのは「物語の進行」と「演じるための文脈」だった |
| 形式化コスト > 恩恵 | 「日記を読むと地下室が開く」がテキスト1行で済むところを多段フォーム入力で構築させた |
| ルート概念の欠落 | 物語の進行構造を設計する概念がなく、シナリオが構築できない |
| 道具の不在 | NPCの人格・口調・知識、モノの所在、分岐条件を「今すぐ使える形」で出す機構がない |
| UIへの過剰投資 | 概念が間違っている段階でUIを磨いた |

**教訓: 実装の正しさは製品価値を保証しない。検証すべきは常に「KPの負担が減るか」。**

## 2. ミッション

> **KPの外部脳になる。** キーパリングに必要な情報量 (多数のNPCの思考トレースと演じ分け、大量のモノ、分岐条件、時限イベント) はKPの脳内に収まらない。ツールはこれらを「瞬時に認識・判断・編集可能な形」で提供し、認知負荷を肩代わりする。

### 確定済みの前提 (ユーザー回答より)

1. **執筆と運用は同一行為。** セッション中、KPは常に執筆している。モード分割しない。
2. **描写は同時に1箇所。** 1ページにまとまったシーン情報を見ながら回せる形が望ましい。
3. **シーン遷移グラフは成立しない。** 進行の基本は自由探索。
4. **自由探索だけでは話が収束しない。** 大筋・ルート管理が別途必要。
5. **タイマー機構は不要。** 時間イベント=条件イベント。
6. **運用時情報に優先順位はない。** 「今現在の事実」を示すものはすべて必須級。
7. **即興編集はすべて同等に重要。** 分岐組み換え・NPC即席投入・手がかり再配置のいずれも頻出。

## 3. コア概念

### 3.0 大原則: 実体とビューの分離

**NPC・モノ・手がかり・事実・真相は独立した実体である。シーンページはそれらの「投影 (ビュー)」であり、保存場所ではない。**

これを破ると、複数シーンに登場するNPCの状態がシーンごとにコピーされ、v5で潰した「片方だけ更新して矛盾」がデータレベルで再発する。シーンページが所有してよいのは: シーン名と種別 (場所/場面/夢など)・PL向け場面描写 (差分含む)・シーン固有のKPメモ・出口/関連シーンリンク・関連する実体へのリンク (投影対象の指定)。**所有してはならないもの**: NPCの現在感情/知識/目的、モノの所在/開示状態、事実の真偽、大筋の達成状態。

### 3.1 シーンページ — 運用の基本単位

1画面 = 1シーン。シーンページには、KPがその場面を回すために必要なすべてが**投影**される:

- 場面描写 (PLに読み上げるテキスト。状況差分も)
- **今ここにいる**NPCのカード (所在の事実から自動投影。§3.4)
- **今ここにある**モノ・手がかり (所在・開示状態付き)
- ここで起きうる条件付きイベント (「リンクされた事実が成立したら → △△」)
- 出口・関連シーン

すべての表示はその場で編集でき、編集は実体に書き戻される (投影なので二重管理は構造上発生しない)。

### 3.2 事実台帳 — 「今現在の事実」の単一の真実源

- **事実 = 短い自由文の宣言** (「執事は死んでいる」「PCは地下室の存在を知った」) + 成立/未成立のトグル + 任意のリンク (NPC/モノ/シーン/真相へ)。
- **条件は事実IDへのリンクの連言で表現する** (機械は自由文を解釈しない。成立ブール値だけを見る)。リンクには**否定フラグ** (「この事実が不成立であること」) を付けられる。条件付きイベント・真相の達成・シーンの解放はすべてこの一つの機構で動く。執筆時の操作は「自由文を書く」「リンクする」の2つだけ — v5の参照型/カテゴリ/値の組み立てを全廃する。
- **排他スロット**: 所在 (モノ・NPCの現在地)、開示状態、NPCの現在感情など「同時に1つしか成立しない」ものはスロットとして扱い、新しい値の成立が古い値を自動で取り消す (「日記は書斎にある」と「日記はAが持っている」の併存を構造的に防ぐ)。
- **定型はツールが負担する** (検証ケース1の最重要知見): 所在・開示スロットはモノ/NPCを作って配置した時点で自動生成され、人が宣言文を書くのは固有の事実のみ。条件付きイベントは**内蔵の発生状態 (未発生/発生済み/抑止)** を持ち、「一度きり」を事実ペアで書かせない。モノ表とシーンページには所在・開示の**現在値が常に投影**される (「日記はどこ?」は1ホップで答える)。
- 事実の成立・取消はワンタッチ。**全変更は履歴に残り、undoできる** (ライブ編集の安全性)。履歴はそのままセッションログの背骨になる。

### 3.3 真相と手がかり網 — 大筋の中身 (収束管理層)

マイルストーンの線形列では「なぜ到達できないか」を診断できない。大筋は**真相 (revelation) のネットワーク**として持つ:

- **真相** = PLに理解させるべき結論 (「地下室に何かがいる」「コービットは死んでいない」)。基本は緩い順序付き (導入→核心→決着) だが、シーンを支配しない。
- 各真相は **複数の手がかり** を持つ。手がかり = 事実+入手導線 (どのシーン/NPC/モノから、どうやって)。重要な真相には複数の導線を用意する (三手がかり原則の支援)。
- **「発見済み」と「理解済み」を区別する。** 手がかりの開示 (事実成立) は自動で追えるが、PLが結論に達したかはKPが判断して真相に「理解済み」を付ける。
- KP画面の常設表示: 「物語はいまどこか」「次の真相に足りない手がかりは何か」「それを渡せるNPC/場所はどこか」。停滞時の導線はメモではなく**投入可能な手段のリスト**として持つ。

### 3.4 NPCカード — 演じるための最小コンテキスト

NPCは独立実体。カードの内容:

- 静的: 人格・動機・口調 (一人称/語尾/態度)・外見・技能値等の数値 (KPに提示するだけ。ダイスは振らない)
- 動的 (= 事実として管理され、カードに投影される): **今の目的/次にやろうとしていること**・恐れていること/隠したいこと・現在の感情・今知っていること・現在地
- KP秘密と読み上げ可能文の区別 (§3.6)

「感情」だけでは演じられない。**意図 (今何をしようとしているか)** が思考トレースの核。

### 3.5 常時編集と安全性 (Writing = Running)

- シーンページ・事実台帳・真相網・NPCカードのあらゆる表示は、見ているその場で追加・修正・削除できる。「編集画面へ移動する」操作を存在させない。
- 同じ実体をどこから編集しても結果は同一 (実体とビューの分離の帰結)。
- **undo・変更履歴は必須。** ライブ中の誤操作が戻せないツールは使えない。
- **走り書きの受け皿**: 構造化が間に合わない瞬間のために、どこでも即メモ→後から事実化/手がかり化/NPC化できる導線を持つ。

### 3.6 KP秘密とPL提示文の区別

すべてのテキストは「PLに出してよい文 (読み上げ・コピー可)」と「KP専用 (真相・秘密・即興メモ)」のどちらかに属する。混在による事故 (秘密の読み上げ) を構造的に防ぐ。

### 3.7 執筆ガイドライン (過剰モデル化の防止 — 検証ケース1の知見)

- **条件には物理的・論理的必然のみを書く。** 「依頼を受けていないと新聞を調べられない」のような動機・正当性ゲートは即興を殺す。動機の判断は常にKPの裁量
- **汎用NPC (司書・受付等) のカード化は任意。** シーンのKPメモ1行で足りるならそれでよい
- **空き部屋にフルのシーンページは不要。** 親シーン内の一行サブ項目として書ける軽量表現を持つ
- **知識の既定はパーティ単位。** 「PCたちは地下室の存在を知った」で始め、別行動などで差分が必要になった時だけPC単位の事実に分割する

## 4. 必須機能要件 (コア概念に付随)

- **検索/ルックアップ**: 「日記はどこ?」「鍵を誰が持っている?」「鈴木は何を知っている?」に即答できること。
- **セッションログ**: 描写・事実成立・イベント発火の時系列記録。ログ項目から事実化/手がかり化できる。
- **パーティ別行動**: PCごとの現在地と知識差分を扱える (同時描写は1箇所でも、別行動は常態)。
- **事前準備の最低ライン**: 市販シナリオを「全部モデル化」しなくても回り始められること。シーン骨子+主要NPC+真相網だけで開始でき、残りはセッション中に書き足す。
- **PLの誤解の扱い**: PLが誤った推理を信じている状態は、真相の「理解済み」が付かないことで表現する (専用機構は作らない)。

## 5. 非ゴール・後続スコープ

- リアルタイムタイマー機構 (時間イベント=条件イベント)
- UIの磨き込み (概念検証まで凍結)
- 自動化の押し付け (機械は事実のブール値評価と通知のみ。判断と宣言は常にKP)
- 後続検討: ハンドアウト管理 (渡した相手/時点)、複数セッション・キャンペーン継続、停滞検出のペーシングシグナル、PL向け機能

## 6. 検証計画 (実装より先)

**題材は公開シナリオ。著作物のためシナリオ本文・モデル化シートは公開リポジトリにコミットしない (local/ で扱う。描写文は自前の言い換えのみ)。**

### ケース (最低3種)

1. **場所探索中心**: 「悪霊の家」(The Haunting) — 街での自由調査+屋敷探索の二相構造。シーンページ/真相網/事実台帳の基本検証 (第1弾)
2. **NPC会話・感情変化中心**: 嘘と説得が主軸の短編 (選定はケース1の結果を見て)
3. **非線形・想定外対応**: 手がかり再配置が必要になる短編

### 方法と合格基準 (測定可能な形)

- **モデル化テスト**: 同じシナリオで「普通のテキストメモ」も作りベースラインとする。比較軸は**「KPが決める項目数」** (シーン・NPC・真相・手がかり・イベント・固有事実の数)。ツールが自動生成する定型 (スロット宣言・発生状態) は数えない。決める項目数がベースラインのメモ内容と同等であること (ケース1で文字量比2.46倍のうち大半が自動生成可能な定型と判明したため、文字量基準は廃止)。
- **紙上ウォークスルー**: 想定外行動を含むPL行動列で進行をシミュレートする。必須イベント例: 重要NPCの無視/殺害、手がかり見落とし、アイテムの想定外移動、PC別行動、NPCの目的変更、セッション中の新NPC追加、核心への飛び級、誤推理の固執、誤操作からのundo、前のシーンへの出戻り。
- 合格基準:
  - 「日記はどこ?」に5秒以内で答えられる (シート目視)
  - 「このNPCは今何を知り、何をしようとしている?」に10秒以内
  - 「次の収束に足りない手がかりと、それを渡せる場所」に10秒以内
  - 想定外行動への対応が既存情報の二重修正なしで完了する
  - 入力した構造化項目のうちセッションで一度も参照しなかったものが2割以下 (過剰モデル化の検出)
- 検証で概念が壊れたら本書を改訂して再検証。**通ってから**データモデル・実装方式 (既存エンジン再利用の可否含む) を設計する。

## 7. 既存資産の扱い

- 現UI: **凍結**。新概念の検証結果が出るまで追加投資しない
- コアエンジン (条件評価・stabilize): 「事実リンク条件の評価+成立通知」の内部実装として**再利用候補**。判断は検証後
- v5仕様書: 歴史的文書として保持

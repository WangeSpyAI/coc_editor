# 要件定義 v6: KPの外部脳

v5仕様の完全実装 (2026-06-13) をユーザーが実際に使用した結果、「使い物にならない。便利でなく負担だけが増える。シナリオを構築できない (ルート設計ができない)」と判定された。本書はその失敗分析と、対話による要件再定義の結果である。**v5とその実装は本書の判断基準としては無効**。コードは資産として凍結し、再利用は概念検証の後に判断する。

---

## 1. v5実装の失敗分析

| 失敗 | 内容 |
|---|---|
| モデル化対象の誤り | ペトリネットは「世界の状態」をモデル化したが、KPに必要なのは「物語の進行」と「演じるための文脈」だった |
| 形式化コスト > 恩恵 | 「日記を読むと地下室が開く」がテキスト1行で済むところを多段フォーム入力で構築させた。トリガー自動連鎖の恩恵は大量の符号化の後にしか発生しない |
| ルート概念の欠落 | シナリオの進行構造 (導入→探索→分岐→収束) を設計する概念が存在せず、そもそもシナリオが構築できない |
| 道具の不在 | KPが現場で必要とする情報 (NPCの人格・口調・知識、モノの所在、分岐条件) を「今すぐ使える形」で出す機構がない |
| UIへの過剰投資 | 概念が間違っている段階でUIを磨いた |

**教訓: 実装の正しさ (テスト・型・構造的安全性) は製品価値を保証しない。検証すべきは常に「KPの負担が減るか」。**

---

## 2. ミッション

> **KPの外部脳になる。**
> キーパリングに必要な情報量はKPの脳内に収まらない (多数のNPCの思考トレースと演じ分け、大量のモノ、分岐条件、時限イベント)。ツールはこれらを「瞬時に認識・判断・編集可能な形」で提供し、KPの認知負荷を肩代わりする。KPはミスや遅延のない進行に集中できる。

### 確定済みの前提 (ユーザー回答より)

1. **執筆と運用は同一行為。** セッション中、KPは常に執筆している。リアルタイムに変化するストーリーへ対応するには、シナリオ全体が常に素早く書き換え可能でなければならない。執筆モード/運用モードの区別を設けない。
2. **描写は同時に1箇所。** キーパリングは基本的に同時に1つの場所しか描写しない。「1ページにまとまったシーン情報」を見ながら回せる形が望ましい。
3. **ただしシーン遷移グラフは成立しない。** シーンからシーンへ、という固定的な流れには必ずしもならない。基本は自由探索で進行する。
4. **自由探索だけでは話が収束しない。** シナリオの大筋・ルートを管理する機能が別途必要 (これが v5 に欠けていた「ルート設計」)。
5. **タイマー機構は不要。** 実時間タイマーは使わない。「時間で発生するイベント」は「条件下で発生するイベント」と同じものとして扱う。
6. **運用時情報に優先順位はない。** NPC・モノ・条件・手がかり — 「今現在の事実」を示すものはすべて必須級。間違えるわけにいかない。
7. **即興編集はすべて同等に重要。** 分岐の組み換え・NPCの即席投入・手がかりの再配置、いずれも頻出。

## 3. コア概念 (提案)

4つの概念で構成する。いずれも「KPの頭の中にある単位」をそのまま写すことを優先し、エンジンの都合を露出しない。

### 3.1 シーンページ — 運用の基本単位

1画面 = 1シーン。シーンは多くの場合「場所」だが、物語的な場面 (「警察署での事情聴取」「夢の中」) でもよい。シーンページには KP がその場面を回すのに必要なすべてが載る:

- **場面描写** — PLに読み上げるテキスト。状況に応じた差分も書ける
- **登場NPCの演技カード** — 人格・動機・口調 (一人称/語尾/態度)・**今知っていること**・現在の感情状態。演じ分けに必要な最小コンテキスト
- **ここにあるモノ・手がかり** — 開示状態付き (未発見/発見済/誰が知ったか)
- **ここで起きうること** — 条件付きイベントのリスト: 「◯◯したら/◯◯が成立していたら → △△が起きる」。時間系イベントもこの形式で表現する
- **出口・接続** — 隣接シーンへのリンク

**すべての要素はその場でインライン編集できる** (前提1)。表示と編集が同じ場所にある。

### 3.2 大筋 (ストーリースパイン) — 収束の管理層

自由探索で発散する物語を収束させるための、シーンとは独立したレイヤー (前提3, 4)。

- **マイルストーンの列**: 例「導入 → 失踪の発覚 → 館への到達 → 真相の核心 → 決着」。基本は線形、分岐も可
- 各マイルストーンが持つもの:
  - **達成条件** — どの事実が成立したら達成か (例: 「PCが日記の内容を知った」)
  - **達成時に開くもの** — 新たに出現するシーン/イベント/情報
  - **収束への導線メモ** — 停滞時にKPが使える誘導手段 (NPCの介入、新情報の投下など)
- KP画面には常時「物語はいまどこか」「次の収束点に何が必要か」が見える
- **大筋はシーンを支配しない。** シーンの順序を固定せず、「事実」の成立だけを見る。自由探索と収束管理を両立させる要

### 3.3 事実台帳 — 「今現在の事実」の単一の真実源

KPが絶対に間違えてはならないもの = 現在の事実 (前提6)。

- **事実 = 短い宣言文。** 「日記は探索者Aが持っている」「執事は死んでいる」「PCは地下室の存在を知った」
- NPCの知識・感情、モノの所在・開示状態、世界の状態変化は、すべて事実として台帳に記録される
- シーンページの条件付きイベントと大筋の達成条件は、この台帳を参照して評価される
- セッションの任意の瞬間に「今の事実」一覧が1画面で確認できる
- 事実の成立・取り消しはワンタッチ。イベント発火やKPの宣言で自動記録され、手でも直せる

### 3.4 常時編集 (Writing = Running)

執筆と運用に同じ操作体系を使う (前提1, 7)。シーンページ・大筋・事実台帳のあらゆる表示は、見ているその場で追加・修正・削除できる。「編集画面に移動する」操作を存在させない。

## 4. 非ゴール

- **リアルタイムタイマー機構** — 時間イベントは条件イベントとして表現する (前提5)
- **UIの磨き込み** — 概念がシナリオを表現できると検証されるまで、見た目・操作性への投資は凍結
- **自動化の押し付け** — 状態の自動連鎖はKPの代行ではなく、あくまで「事実の記録の手間を減らす」「見落としを通知する」支援に限る
- **PL向け機能** — 対象ユーザーはKPのみ

## 5. 検証計画 (実装より先)

**題材: 市販・公開シナリオ (ユーザー指定)。** 著作物のため、シナリオ本文・モデル化シートは公開リポジトリにコミットしない (ローカルの gitignore 対象フォルダで扱う)。

1. **モデル化テスト**: 題材シナリオを本書の概念 (シーンページ/大筋/事実) で markdown のシートに書き起こす
   - 合格基準: シナリオの全シーン・全NPC・大筋が欠落なく表現でき、書き起こしの手間が「普通にテキストでシナリオメモを書く」場合の2倍以内
2. **紙上セッションウォークスルー**: PLの行動列 (想定外行動を含む) を想定し、シートだけでセッションを進行してみる
   - 合格基準: 任意の時点で「今の事実」「今のシーンで必要な情報」「次の収束点」が即座に答えられる / 想定外行動への書き換えがその場の追記で済む
3. 検証で概念が壊れたら本書を修正して再検証。**通ってから**データモデルと実装方式 (既存エンジンの再利用可否を含む) を設計する

## 6. 既存資産の扱い

- 現UI (場面ビュー・LiveEditor等): **凍結**。新概念の検証結果が出るまで追加投資しない
- コアエンジン (条件評価・stabilize・カテゴリ値): 事実台帳と条件付きイベントの内部実装として**再利用候補**。判断は検証後
- v5仕様書・実装・テスト: 履歴として保持 (docs/spec/trpg_scenario_editor_v5.md は歴史的文書)

# ユースケース起点のKP困難点 → ツール要件 → v6実装状態 (2026-06-16)

UI設計より先にやるべきだった土台。公開シナリオ「悪霊の家」を題材に、各フェーズの **PL行動 → KP対処 → KPの困難点** を網羅列挙し（全列挙は local/the-haunting-usecases.md、非コミット）、そこから繰り返し現れる **KP困難点の類型** を抽出。各類型を **必要なツール操作** に対応づけ、現v6 がそれを満たすかを判定する。

これにより「完成」を「画面が建っているか」ではなく **「KPの実際の困難が操作で解けているか」** で定義し直す。

## 1. KP困難点の類型（実プレイから抽出。CoCキーパリング一般に通用）

## 3. 結論 — 「完成」の再定義と現在地

**v6は、KPの困難の「参照・診断」側はかなり解けている。** 解けている(◯): P3(事実の一括確定/トグル)、P12(理解済み追跡)、P14(Undo)。表示として支えている(△の参照側): 今の事実・収束に足りない手がかりとルート・NPC情報(秘密分離)・読み上げ。これらは実プレイの高頻度の痛みに正しく対応している。

**しかし「統制・操作」側、それも実プレイが最も要求する操作が欠落している:**
- **手がかりの開示統制（P1の中核 = CoCキーパリングそのもの「いつ誰に手がかりを出すか」）** — 表示のみ。
- **見落とし手がかりの再配置（P6）** — 診断のみ。しかも既存の救済(手動フラグ)は CLAUDE.md が禁じる「気をつけるに依存する設計」そのもの。
- **パーティ分割の知識差分（P7、単独最高負荷）** — UI未実装。
- **NPCの動的状態（意図/感情/知識）の場面からの更新（P10）** — 表示のみ。
- **画面間リンク（P2/P6/P8 横断 = 不足手がかりをクリックして「どこで・どう渡せるか」へ飛ぶ）** — 無い。

**さらに第2階層の根本的な発見（モデル自体の限界、P9・P5）:**
- **データモデルが「想定外プレイで生じる状態」を持てない。** 脱線21件の多くが、モデルに無い状態（肉体の持ち出し、刃物の武器化、NPCの移動、鍵の複製、第三者の雇用、台帳に無い場所）をKPが即興で作る羽目になる。モデルは概ね想定ルートを前提にしており、脱線対応がツールでなくKPの即興に丸投げされる。
- **CoCの数値（技能値・SAN・ダメージ・判定）がモデルに無い。** CoC向けKP支援ツールとして実務上の穴。

→ したがって **「完成」とは「KPの上記の困難が操作で解ける」こと。** 現v6は参照・診断の半分＋一部操作（イベント発火→読み上げ・事実トグル・Undo・収束診断）まで。統制の半分（手がかり統制・再配置・分割知識・NPC動的状態・画面間リンク）と、モデルの脱線/数値カバレッジが未完成。**「データは見えるが、実プレイで一番やりたい操作ができない」から試作品に見える。あなたの「未完成」判断は正しい。**

## 4. 次にどこを完成させるか（このユースケース分析が示す優先順）

困難点の頻度・負荷とCoCキーパリングの中核度で優先:

1. **手がかりの開示統制（P1中核）** — 場面/真相から手がかりを「発見させた/共有した/未開示に戻す」ワンタッチ操作。CoCの中核で、現状の最大の穴。
2. **画面間リンク（P2/P6/P8横断）** — 不足手がかり・ルート・ログ・場面カードをクリックで該当箇所へ。「6つの別表」を「繋がった外部脳」にする。
3. **見落とし手がかりの横断一覧＋再配置（P6）** — 「今このパーティが取りこぼしている手がかり」を一覧し、別シーンへ置き直す操作（記憶依存の救済を構造化）。
4. **パーティ分割の知識差分UI（P7）** — コアにある分割/合流をUIに出す。
5. **NPC動的状態の場面からの更新（P10）** — 意図/感情/知識/所在をその場で。
6. **(モデル拡張) 脱線状態の表現力・CoC数値（P9/P5）** — 抽象スロット値の即席付与、技能/SANの保持。上流の判断を要する。

ただし優先順を確定する前に、この **困難点の類型と要件マッピング自体が妥当か**（実際のKP経験と合っているか）をユーザーに確認してから着手する。

# v6 画面設計と実装状態 (2026-06-16)

現行 v6 最小UI(src/ui-v6/)を、各画面ごとに「設計／ユースケース／KP操作／解決する問題／実装状態」で整理した設計説明。実装状態は現コードに照らし、各操作が実在するかを独立検証した結果（complete=ユースケースが操作まで完結 / partial=表示はあるが主要操作が欠落 / missing=表示のみ）。

---

## 0. このツールは何か（全体設計）

**KPの外部脳。** CoCのセッション中、KPの頭に収まりきらない「今の事実・場面・登場NPC・手がかり・真相・今発火できるイベント」を画面群で外部化し、認知負荷を肩代わりする。

- **中核モデル: 事実(Fact)が単一の真実源。** 場面・イベント・真相・NPC状態は、事実を参照/変更する「投影」。所在・開示・感情などは排他スロットの事実。
- **設計上のKPセッションループ:**
  1. 場面を開く → PL描写を読み上げ、NPCを演じる
  2. プレイヤーが動く → 事実を更新 or イベントを発火（→読み上げ描写が出る）
  3. 詰まる → 真相一覧で「理解されてない真相・足りない手がかり・渡せるルート」を見る
  4. 「今どうなってる?」 → 検索で即答（鍵はどこ/鈴木は何を知っている）
  5. 経緯を追う → セッションログ
  6. 誤り → Undo/Redo
- **6ビュー + 場面サイドバー + ヘッダ**で構成。

## 1. 画面別 設計カード

### A. 全体フレーム/導線（オンボーディング・ヘッダ/ナビ・シーン一覧・Undo/Redo）
- **設計**: 6ビューを切り替える土台。初見で「これは何か」を示し、場面を選び、誤りを取り消す。
- **ユースケース**: 起動直後（案内）→ セッション中ずっと、ビュー切替と場面選択の起点。
- **KP操作（実在）**: 案内の開閉(サンプルで試す/？使い方/Esc)・シーン選択・ビュー切替・Undo・Redo・demo読込。
- **解決する問題**: 多数の情報源を一箇所から素早く参照。初見の立ち上げ。誤操作の取り消し。
- **実装状態: partial** — 土台の操作は揃う。欠落: **画面間リンク（場面カードや真相のルートをクリックしても飛べない）／新規シーン追加UIなし／シナリオのインポート・エクスポートUIなし（メソッドは在るが呼び口なし）／タブ件数バッジはイベントのみ。**

### B. Scene投影ページ（場面の司令塔）
- **設計**: 現在の場面に必要な全部（読み上げ・NPC・手がかり・発火イベント・出口）を1ページに集約し、文脈切替なしで回す。
- **ユースケース**: 場面に入っている間ずっと開く。読み上げ→演じ→発火→移動。
- **KP操作（実在）**: ①今読み上げる描写の表示+コピー ②PL描写編集(保存) ③KPメモ編集(保存) ④**イベント発火(発生させる)→読み上げ描写がその場に出る** ⑤出口で場面移動(進入条件未達はdisabled)。
- **解決する問題**: シナリオ文書を行ったり来たりせず、今の場面の状態と「次に起こせること」が一目で分かる。
- **実装状態: partial** — 読み上げ・編集・発火・移動は動く。**欠落（重要）: 手がかりの開示統制（発見させる/共有がなく表示のみ）／NPCの動的状態（意図・感情・知識・所在）の変更不可（表示のみ）／関連真相は表示のみ。** = 「場面で起きたことを記録・統制する」操作が薄い。

### C. 事実台帳
- **設計**: KPのリアルタイム真理値表。事実が分岐・知識・イベント発火を駆動する単一の真実源。
- **ユースケース**: プレイヤーの発見/選択で事実が確定したとき、即席で事実を足すとき、なぜ発火しないか調べるとき。
- **KP操作（実在）**: 事実のtrue/falseトグル・新規事実の作成・メタ表示(出所/ID/スロット束縛)。
- **解決する問題**: 20以上の事実を頭で追う代わりに、一覧で現在値が見え、ワンタッチで切替。KPの内部モデルと系の状態を同期。
- **実装状態: complete** — ユースケースが操作まで完結（唯一の完成画面）。

### D. 真相一覧
- **設計**: 物語の収束管理。各真相の理解状況・未発見手がかり・不足事実・渡せるルートを可視化。
- **ユースケース**: 停滞時/収束確認時に「何が伝わっていて、何が足りないか」を診断。
- **KP操作（実在）**: 理解済みチェックのトグル。（発見済み/未発見手がかり・不足Fact・利用可能ルートは表示のみ）
- **解決する問題**: 「どの真相が伝わったか」を手帳でなくライブデータで把握。停滞を即診断。
- **実装状態: partial** — 理解済みトグルは動く。**欠落: 手がかり名をクリックしても飛べない（プレーンテキスト）／ルートをクリックしても該当場面に行けない／編集・並び替え・一括操作なし。** = 「診断はできるが、そこから動けない」。

### E. イベント通知
- **設計**: 今発火できるイベントと「あと1条件」のイベントを示すリアルタイム警報。
- **ユースケース**: 今使えるプロットフックの確認、次に押すべき事実の発見、手動発火。
- **KP操作（実在）**: 発火可能イベントの発火(発生させる)。
- **解決する問題**: どのイベントが今使えるか/あと何が要るかを覚えなくてよい。
- **実装状態: partial** — 発火は動く。**欠落: 現在の場面でフィルタされず全イベント表示／発火前に結果(どの事実が変わるか)を予見できない／発火済み履歴なし／抑止(veto)UIなし。**

### F. セッションログ
- **設計**: 読み上げ・事実変化・イベント・スロット変更の時系列ジャーナル + KPの手動メモ。
- **ユースケース**: 「さっき何の事実を立てた?」「いつあのNPCが動いた?」を後から追う。
- **KP操作（実在）**: KPメモの追加。
- **解決する問題**: セッションの経緯を失わない外部記憶。
- **実装状態: partial** — メモ追加は動く。**欠落: ログ項目は表示のみ（編集/削除/クリックで該当事実へ飛ぶ等なし）／フィルタ・検索・書き出しなし／（設計に「読み上げを自動ログ」とあるが現状は自動キャプチャされない＝設計と実装の乖離）。**

### G. 検索
- **設計**: 事実/NPC/モノ/手がかり/場面の即時ルックアップ。「鍵はどこ/鈴木は何を知っている」に即答。
- **ユースケース**: 想定外の質問・行動で、キーワードから該当実体を引き、クリックでその詳細へ飛ぶ。
- **KP操作（実在）**: クエリ入力・結果表示(一致種別バッジ付き)・**結果クリックで該当ビューへ遷移**（場面系→Scene、真相→真相、イベント→イベント）。
- **解決する問題**: タブを3〜5回行き来せず、1クエリ+1クリックで文脈ごと着地。
- **実装状態: partial（ほぼcomplete）** — 検索とクリック遷移は動く。軽微な欠落: 事実(orphan)へ飛ぶと台帳上で該当事実がハイライト/スクロールされず埋もれる。

## 2. 正直な総括 ―「未完成に見える」の正体

あなたの直感は**半分当たっている**。「全部未完成」ではない（事実台帳は完成、参照系の表示は概ね動く）。が、**操作系が薄く、画面が繋がっていない**ため、試作品に見える。

| 画面 | 検証済み実装状態 |
|---|---|
| 事実台帳 | **complete** |
| 検索 | partial（ほぼ完成、飛び先ハイライトのみ欠落） |
| Scene投影 | partial（読み上げ/編集/発火/移動○、手がかり統制・NPC状態変更×） |
| 真相一覧 | partial（理解済み○、リンク・編集×） |
| イベント通知 | partial（発火○、絞り込み・結果予見・履歴×） |
| セッションログ | partial（メモ追加○、編集/フィルタ/リンク×） |
| 全体フレーム | partial（土台○、画面間リンク・新規シーン・入出力UI×） |

**構造的な欠落は3つ:**
1. **手がかりの開示統制がない** — CoCキーパリングの中核は「手がかりをいつ誰に出すか」。今は表示のみで、KPが場面から「発見させた／共有した」を操作できない。
2. **NPCの動的状態を場面から変えられない** — 意図・感情・知識・所在が表示専用。事実台帳でトグルする必要があり、場面の流れで操作できない。
3. **画面が相互リンクしていない** — 真相のルート・ログ項目・場面のカードをクリックしても飛べない。だから「6つの別々の表」に見え、「外部脳」として繋がっていない。

つまり今のv6は「**よくできた参照ダッシュボードの骨格 + 一部の操作**」であって、「KPが操作で世界を統制する道具」にはまだ届いていない。

## 3. 次の判断（あなたに問う設計レベルの選択）

- **(a) 操作系を完成させる**: 上の欠落3つ（手がかり統制・NPC状態操作・画面間リンク）を埋め、参照ダッシュボードを「統制できる外部脳」にする。中核は手がかり統制。
- **(b) モデル/構成自体を問い直す**: そもそも「6タブ + 事実駆動」がKPの実運用に合うのか、上流に戻って再検討する。
- **(c) 別の優先**: 実シナリオでの通し運用テストを先にやる、等。

設計者としての推奨は **(a)** だが、まずこの設計説明自体への評価（設計意図は妥当か、ユースケースは合っているか）を聞きたい。

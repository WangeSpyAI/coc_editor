# 統合と完成までの設計 (2026-06-12)

## 背景: 2系統の存在と統合判断

このプロジェクトには2つの系統が存在していた。

| 系統 | 内容 | 状態 |
|---|---|---|
| ローカル (trpg_scenario_editor) | v5概念設計書 + プロトタイプv8b (603行JSX) + TS化設計/実装計画 | TS化はほぼ未着手 (型定義のみ、2026-03) |
| GitHub (coc_editor) | ペトリネット意味論ベースの別実装 (58コミット) | React+TS、エンジン純粋関数化、テスト、undo、localStorage永続化、GitHub Pagesデプロイ済 (2026-04) |

**判断: リモート実装 (coc_editor) をベースに採用。**

- v5仕様カバー率はリモート約75% vs プロトタイプ約60% (かつJS・テストなし)
- リモートには型安全性 (ReadonlyWorldState、MutationAPI)、Vitestテスト、undo、デプロイ基盤が既にある
- 残作業はリモートの不足分実装 (数日相当) vs ゼロから構築 (数週間相当)

### リポジトリ統合の記録

- coc_editor のデフォルトブランチ `claude/coc-scenario-editor-repo-6ZMw1` を `main` に改名
- ローカル資産を docs/ に統合: v5仕様書 → `docs/spec/`、プロトタイプ → `docs/prototype/`、旧TS化計画 → `docs/plans/archive/` (採用されなかった経路の記録)
- ローカル履歴は `archive/local-prototype` ブランチとして coc_editor に保全
- ローカルの feature/ts-refactor ブランチは破棄 (型定義のみで、リモート実装に劣る)

## 要件 (確定)

**ゴール: 実際にキーパリングとシナリオ執筆に使える段階。**

技術判断はすべて開発側に委任された。判断基準は v5 の設計思想:

- 常に編集可能。即興を阻害しない
- KPの操作速度が最優先。認知負荷と操作コストの最小化
- トリガーは支援であり制約ではない
- 場面中心。描写が成果物

## 衝突の扱い

リモート実装は v5 より後発であり、進化分 (地図ビュー、Entity.labels、モバイル対応、構造的安全性ルール) は維持する。v5 は不足機能のチェックリストとして使う。

主な衝突点と裁定:

| 衝突点 | v5 | リモート | 裁定 |
|---|---|---|---|
| ロール判定 | KPが判定し、成功ならボタンを押す。システムは強制しない | エンジン内で成功/失敗分岐 (successEffects/failureEffects) | **折衷**: エンジンの成功/失敗効果は維持しつつ、判定結果の入力はKPが行う (ダイスを自動で振らない、KPが成功/失敗を選ぶ)。v5の「システムは判定を強制しない」を満たす |
| UIレイアウト | 3ペイン (ツリー/場面/右パネル) | 2カラム (1エンティティ=1パネル) | **KPの操作速度で判断**: 場面ビューに必要な要素 (子孫アクション集約、状態ドロップダウン、ミニログ、待機トリガー) が1画面で見えることが本質。実機確認の上、不足があれば v5 レイアウトに寄せる |
| 居場所カテゴリ | 暗黙の排他カテゴリとして全エンティティが持つ | parentId直接操作 | **リモート方式を維持**: parentId が単一の真実。アクション効果で移動を表現できる moveEntity 系 Effect があれば v5 の目的 (アクション結果で移動を表現) は満たせる |

## 完成スコープ

### キーパリング必須

1. 静的描写: カテゴリ値に描写テキストを持たせる
2. 描写合成 (project): 場面 = 自身+子孫の描写を持つ値をカテゴリ表示順で合成
3. 描写ログ強化: コピーボタン、場所移動時の場面描写自動記録
4. パーティ実装: アクティブパーティ、場所ナビゲーション=パーティ移動、分割/合流、切替でビュー追従
5. 進入条件: 条件未充足の場所は🔒で移動不可
6. 待機中トリガーUI: 残り1条件のトリガー表示 + [付与 ▶] ワンクリック発火
7. 行為者選択: PLアクションの行為者選択、必要アイテム/知識による候補絞り込み
8. $actor: Effect の対象/値での置換 (描写は実装済み)
9. 情報共有: 同一場所のPC間で知識カテゴリ値をコピー
10. 振動警告: stabilize 上限到達時のUI警告

### シナリオ執筆必須

11. テンプレート: 種別ボタン (場所/NPC/PC/アイテム/空) でワンステップ作成、初期カテゴリ適用
12. 効果のGUI編集: エンティティ・カテゴリ・値をドロップダウンで選択
13. エンティティ削除 (子孫・参照の整合性処理込み)
14. ファイル保存/読み込み: JSON エクスポート/インポート (実装済みなら動作検証)

### スコープ外 (後続フェーズ)

- 依存グラフの拡充 (既存 DependencyGraph.tsx の動作確認まで)
- CoC的パラメータ (技能値シート)。ロール条件テキスト表示で代替
- 複数undo履歴のブランチ化

## 進め方

1. リモート実装の実機検証 (build/test/dev起動) で現状を確定
2. 実装計画を docs/plans/ に作成、タスク分割
3. サブエージェント駆動で実装、各タスクにコードレビュー
4. CLAUDE.md の構造的不変条件 (状態更新3プリミティブ、ReadonlyWorldState、applyEffect経由) を全タスクで厳守
5. 完成判定: サンプルシナリオで「執筆→セッション進行→ログ出力」の一連の流れを通しで検証

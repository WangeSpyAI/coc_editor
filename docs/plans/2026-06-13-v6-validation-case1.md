# v6.1 概念検証 ケース1: 場所探索型シナリオ (悪霊の家)

公開シナリオ「悪霊の家」(The Haunting) を v6.1 概念でモデル化し (local/ 配下、非コミット)、同内容の通常テキストメモと比較、想定外行動を含むウォークスルーで評価した。

## 計測

| 項目 | 値 |
|---|---|
| モデル化シート | 21,410字 |
| 通常メモ (ベースライン) | 8,713字 |
| 比率 | **2.46倍 (基準2倍を超過)** |
| 構造化項目 | 事実71 / シーン22 / NPC8 / 手がかり24 / イベント25 / モノ実体21 / IDリンク372 |

## 判定: 概念は成立。膨張は「手で書く定型」が主因で、機構で吸収可能

### 成立した点

- **実体とビューの分離は機能した。** NPC/モノ状態の二重記述はゼロ。シーンはリンクのみ持ち、状態矛盾が構造的に起きない
- **真相網は収束管理として機能した。** 真相7件それぞれに複数手がかり+導線が並び、「単一手がかりのボトルネック」(隠し区画の発見、決着手段) が執筆時点で可視化された — これは通常メモでは得られない成果
- **事実リンク条件は監査可能だった。** 怪異エスカレーション・最終対決の発火条件が事実IDの連言で追える。KP判断の「再配置」(f26型の宣言事実) も自然に表現できた
- 表現不能なものは無かった (codexモデル化担当・私のレビュー双方で確認)

### 発見された問題 (v6.2で対処)

1. **イベント発生状態を事実ペアで書かせるのは機構の欠陥。** 「未発生/発生済み」の排他ペアが8組16事実を占めた。イベント実体が内蔵の発生状態を持てば消える
2. **単一値スロットの定型行が膨張の最大要因。** 「Xは初期位置にある。初期=成立」型の行が約30件。これはモノ/NPCを作って配置すれば自動生成されるべき情報で、人が書く文ではない。**計測も「文字量」でなく「KPが決める項目数」で行うべき** — 決める量で比較すればベースラインとほぼ同等だった
3. **条件の「動機ゲート」汚染。** 街調査イベントの多くが「依頼を受けた(f2)」を条件にしており、依頼を飛ばす即興 (鍵を盗む等) で全調査イベントが死ぬ。執筆ガイドラインが必要: **条件には物理的/論理的必然のみを書く。動機・正当性はKPの裁量**
4. **汎用NPCのカード化は過剰。** 司書・受付等4枚は口調1行で足りる。カード化は任意とする
5. **空き部屋にフルのシーンページは過剰。** 親シーン内の一行サブ項目で足りる軽量表現が要る
6. **PC個人の知識が未モデル。** 事実が「PCは〜を知った」とパーティ単位になっており、別行動時の知識差分が表現できない。既定はパーティ単位 (軽い)、必要時にPC単位へ分割できる形にする
7. **条件に否定 (不成立要求) が無い**ため、「まだ〜していない」を排他ペアで偽装した。リンクに否定フラグを許せば解決
8. (紙ゆえの制約) ルックアップ「日記はどこ?」はモノ表→所在スロットの2ホップが要り5秒を超えうる。ツールでは現在値をモノ表・シーンページに投影するため解消する見込み。**ツール実装時の必須投影として記録**

### ウォークスルー抜粋 (想定外行動)

- 依頼を飛ばして鍵を盗む → 所在スロット書き換え1回で進行可。ただし問題3により街調査が死んだ (ガイドラインで解決)
- 手がかり見落とし → KPの宣言事実+再配置イベントで二重修正なしに復旧
- 家主を殺害 → 所在スロットに「死亡」値を追加成立するだけ。以後の投影から消える
- 誤操作undo → 事実トグルの履歴で戻せる (機構要件として既定義)

## v6.2への反映 (要件文書を改訂)

機構側: イベント内蔵発生状態 / 所在・開示スロットの自動生成 / 条件リンクの否定フラグ / 現在値の投影 (モノ表・シーンページ)。
執筆ガイドライン: 条件は必然のみ / 汎用NPCカード任意 / 軽量サブ項目 / 知識は既定パーティ単位。
計測: 「KPが決める項目数」基準に変更。
